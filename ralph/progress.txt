## Codebase Patterns
- Settings table uses inline CREATE TABLE in init_db() (not migration files) since it's a simple key-value table
- Backend handlers in main.rs follow pattern: async fn handler(State(state): State<AppState>, ...) -> Response with match on Result
- Frontend API client uses typed namespace objects (e.g., `settings`, `profiles`) with method functions calling `request<T>(path, options, requiresAuth?)`
- SQLite upsert uses INSERT ... ON CONFLICT(key) DO UPDATE SET ... pattern
- Auth is frontend-only (Bearer header via requiresAuth flag); no backend middleware yet
- Svelte 5 rune stores use `.svelte.ts` extension with exported `$state` objects; pure functions exported separately for testability
- Profile/chart types (`ProfileWithPoints`, `ProfilePoint`) live in `$lib/api/client.ts` â€” import from there
- Chart.svelte exposes `getInstance()` for attaching ECharts event handlers (getZr click, series click)
- ECharts `convertFromPixel('grid', [offsetX, offsetY])` converts pixel click coords to data coordinates
- Use `getZr().on('click')` with `!event.target` check for blank area clicks (add points)
- Use `chart.on('click')` with `componentType === 'series'` for clicking on data points
- Svelte 5 `$props()` values referenced outside reactive contexts trigger `state_referenced_locally` warning â€” extract to plain `const` variables to make intent clear
