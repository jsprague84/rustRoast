## Codebase Patterns
- Migrations use `include_str!("../migrations/xxx.sql")` with split-by-semicolon execution and silent error handling (tables may already exist)
- Use `IF NOT EXISTS` for all CREATE TABLE/INDEX/TRIGGER statements for idempotency
- Triggers use `CREATE TRIGGER IF NOT EXISTS` syntax for updated_at auto-update
- Foreign keys use `ON DELETE CASCADE` for child tables, `ON DELETE SET NULL` for optional references
- AppState is in main.rs with clone-able fields (Arc<RwLock<...>>, SqlitePool, etc.)
- RoastSessionService follows the service pattern: struct with SqlitePool, async methods returning Result
- Route handlers are defined in main.rs (monolithic file) — control, session, profile, telemetry handlers all inline
- Models (request/response structs) are in models.rs with Serialize/Deserialize derives
- Tests use in-memory SQLite with the same migration pattern
- serde_json::Value in FromRow structs requires: sqlx `json` feature in Cargo.toml + `#[sqlx(json)]` attribute on the field
- When binding serde_json::Value to queries, convert to string first: `serde_json::to_string(&value)?`
- set_register_map uses a transaction: DELETE all existing + INSERT new entries for atomic replacement
- Route handlers can be separated into `routes/*.rs` files and merged via `Router::merge(device_routes())`
- AppState must be `pub(crate)` for route modules to extract it; service fields also need `pub(crate)`
- AppError pattern: struct with StatusCode + message, impl IntoResponse to emit JSON `{ error, status }`, impl From<anyhow::Error>
- Existing `/api/devices` was renamed to `/api/devices/registry` (in-memory MQTT device cache); new `/api/devices` serves persistent device config
- Standalone server components (Modbus TCP) go in `src/modbus.rs` (not services.rs which is for DB services)
- tokio-modbus 0.16 TCP server needs `features = ["tcp-server"]`; Service trait `call(&self)` requires Pin<Box<dyn Future>> for async ops
- When starting background tasks that clone Arc'd state, order matters: clone before the consumer loop that takes ownership
- mqtt_consumer_loop uses `#[allow(clippy::too_many_arguments)]` since it legitimately needs many Arc'd caches plus services
- Auto-discovery creates device + MQTT connection in two calls; if connection creation fails, device still exists (partial success is acceptable)
- Debounce HashMap<String, Instant> tracks per-device last-seen update times; no Arc needed since it's local to the consumer loop
- Axum 0.7 (matchit) correctly handles literal routes like `/api/devices/discovered` alongside parameterized `/api/devices/:id` — literal takes priority

## 2026-02-28 - DEV-006
- Integrated Modbus TCP server into the backend using tokio-modbus 0.16 tcp-server feature
- Server is disabled by default, enabled via RUSTROAST_MODBUS_ADDR env var (e.g. "0.0.0.0:502")
- Default device_id configurable via RUSTROAST_MODBUS_DEVICE_ID env var (defaults to "esp32-001")
- Input registers (read-only): 0x0000-0x0001 bean_temp (float32), 0x0002-0x0003 env_temp (float32), 0x0004-0x0005 rate_of_rise (float32), 0x0006 heater_pwm (uint16), 0x0007 fan_pwm (uint16)
- Holding registers (read-write): 0x0000-0x0001 setpoint (float32), 0x0002 fan_pwm_setpoint, 0x0003 heater_pwm_setpoint, 0x0004 control_mode, 0x0005 heater_enable, 0x000C emergency_stop
- Writes to holding registers publish corresponding MQTT control commands via MqttService
- Each TCP connection gets its own Service instance via factory closure; all share Arc'd state with Axum server
- Files changed:
  - `crates/server/Cargo.toml` — Added `tcp-server` feature to tokio-modbus dependency
  - `crates/server/src/modbus.rs` — New: RoasterModbusService implementing tokio_modbus::server::Service, ModbusSharedState, input/holding register read/write logic, MQTT command publishing, start_modbus_server() entry point
  - `crates/server/src/main.rs` — Added `mod modbus`, call to `modbus::start_modbus_server()` before MQTT consumer loop (to clone telemetry_cache before it's moved)
- **Learnings for future iterations:**
  - tokio-modbus 0.16 TCP server requires `features = ["tcp-server"]` in Cargo.toml — the default features only include the TCP client
  - The Service trait's `call(&self)` takes immutable reference; use Pin<Box<dyn Future + Send>> as the Future type to perform async operations (reading from tokio::sync::RwLock, publishing MQTT)
  - Modbus TCP server is a standalone module (modbus.rs) rather than part of services.rs, since it's a server component not a database service
  - When starting Modbus server before mqtt_consumer_loop, clone telemetry_cache first since mqtt_consumer_loop takes ownership
  - Float32 values are encoded IEEE 754 big-endian across 2 consecutive 16-bit registers: high word at lower address, low word at higher address
  - Holding registers 0x0006-0x000B are gap/reserved registers (no MQTT mapping) — reads return 0, writes are no-ops
