## Codebase Patterns
- Control handlers in control.rs return `Result<Response, AppError>` ‚Äî use `AppError::Validation` for input range checks
- `publish_qos1` in state.rs returns `Result<Response, AppError>` using `AppError::MqttUnavailable`/`MqttTimeout`
- AppError has `From<JsonRejection>` impl ‚Äî malformed JSON bodies automatically produce structured error responses
- All API errors should use the shape: `{ "error": "<type>", "message": "<detail>" }`
- Pre-existing a11y warnings: 8 warnings in ControlPanel/PidControls (MOD-011 scope)
- Session/profile route handlers now return `Result<Response, AppError>` ‚Äî use `AppError::NotFound` for missing resources
- Use `AddTelemetryPointParams` struct instead of individual args for `add_telemetry_point`
- Config struct only contains app-level settings; MQTT/DB config loaded by their respective modules
- DB migrations use `sqlx::migrate!("./migrations")` ‚Äî add new `.sql` files to `crates/server/migrations/` with format `YYYYMMDDHHMMSS_description.sql`
- Use `IF NOT EXISTS` and `INSERT OR IGNORE` in migration files for idempotency with existing databases
- Test helper `test_db()` also uses `sqlx::migrate!()` ‚Äî no manual SQL setup needed for tests
- Use `rustroast_server::sqlite_options(url)` to build `SqliteConnectOptions` with all production PRAGMAs ‚Äî shared between main.rs and tests
- SQLite PRAGMAs must be set via `SqliteConnectOptions` (not ad-hoc queries) so every pool connection gets them
- HTTP integration tests use `test_app()` helper (no auth) and `test_app_with_auth(key)` (auth enabled): builds full Axum router with in-memory SQLite + `MqttService::mock()`, clone the Router for multi-request tests
- Axum layer ordering matters: `Extension` must be the outermost layer (applied last) so inner middleware can read request extensions
- `MqttService::mock()` creates a disconnected MQTT service with `is_ready()` returning true ‚Äî use for HTTP tests that don't need real MQTT
- lib.rs exports `auth`, `error`, `models`, `routes`, `services`, `state`, `telemetry` ‚Äî integration tests import from `rustroast_server::*`
- `tower::ServiceExt::oneshot()` pattern: `app.clone().oneshot(request).await` ‚Äî Router must be cloned between requests since oneshot consumes it
- Use `http_body_util::BodyExt::collect()` to read response bodies in tests
- WebSocket handlers use `tokio::select!` with 3 branches: ping interval (30s), client messages, MQTT events ‚Äî skip first interval tick before the loop
- Broadcast `RecvError::Lagged(n)` should warn + continue (resume from latest), `RecvError::Closed` should error + break ‚Äî never use `Err(_) => {}` or `Err(_) => break` for broadcast receivers
- Broadcast channel capacity is 1024 (in `MqttService::connect()` and `MqttService::mock()`) ‚Äî sufficient for ~17 min of 1Hz telemetry burst tolerance
- Background tasks (MQTT consumer, retention cleanup) accept a `CancellationToken` and use `tokio::select!` with `cancel.cancelled()` for graceful shutdown
- AppState caches are grouped in `CacheState` sub-struct ‚Äî access via `state.cache.telemetry`, `state.cache.autotune_status`, etc.
- `mqtt_consumer_loop` takes `AppState` directly (not individual fields) ‚Äî clone state before passing to background tasks
- tower-http layers applied after `.with_state()`: CompressionLayer (innermost) ‚Üí TimeoutLayer (30s) ‚Üí TraceLayer (outermost) ‚Äî last `.layer()` call is outermost in Axum
- Svelte control components use a non-reactive `let synced = false` flag in `$effect` to sync from telemetry only once (first mount) ‚Äî prevents 1Hz updates from overwriting user input
- ECharts uses tree-shaken modular imports in Chart.svelte ‚Äî import from `echarts/core`, register components with `echarts.use()`, export `ECOption` type for consumers
- Landmark constants (colors, labels, emojis) live in `$lib/constants/landmarks.ts` ‚Äî import from there, never define inline
- Session detail types: `SessionWithTelemetry` (extends `RoastSession` + telemetry array) and `SessionTelemetryPoint` (snake_case fields from SQL) live in `$lib/types/session.ts` ‚Äî distinct from WS `Telemetry` type (camelCase from ESP32)
- WebSocket store uses `pausedByVisibility` flag to suppress `onclose` reconnect when disconnecting intentionally (visibility pause) ‚Äî always check this flag pattern when adding new deliberate disconnect paths
- Error notifications: use `notifyError('prefix')` in `.catch()` chains, `notifications.add(msg, 'error')` in try/catch blocks ‚Äî import from `$lib/stores/notifications.js`
- Frontend tests: `vitest run` with jsdom env and `$lib` alias in vitest.config.ts ‚Äî mock WebSocket with `simulateOpen()`/`simulateClose()`/`simulateMessage()` helpers, mock fetch with `vi.stubGlobal('fetch', vi.fn())`
- WebSocket mock cleanup: separate `close()` (silent, for `destroyTelemetrySocket()`) from `simulateClose()` (fires `onclose`, for tests) to avoid dangling reconnect timers across tests
- Svelte components use `interface Props { ... }` for prop typing ‚Äî no need to export types, destructure with `let { ... }: Props = $props()`
- Data-fetching pages use `+page.ts` load functions with `PageLoad` type ‚Äî page components receive data via `let { data }: PageProps = $props()`, errors thrown via `error()` from `@sveltejs/kit` are caught by `+error.svelte`
- After mutations (delete, import), use `invalidateAll()` from `$app/navigation` to re-run load functions ‚Äî data prop updates reactively without manual loading state
