## Codebase Patterns
- Settings table uses inline CREATE TABLE in init_db() (not migration files) since it's a simple key-value table
- Backend handlers in main.rs follow pattern: async fn handler(State(state): State<AppState>, ...) -> Response with match on Result
- Frontend API client uses typed namespace objects (e.g., `settings`, `profiles`) with method functions calling `request<T>(path, options, requiresAuth?)`
- SQLite upsert uses INSERT ... ON CONFLICT(key) DO UPDATE SET ... pattern
- Auth is frontend-only (Bearer header via requiresAuth flag); no backend middleware yet
- Svelte 5 rune stores use `.svelte.ts` extension with exported `$state` objects; pure functions exported separately for testability
- Profile/chart types (`ProfileWithPoints`, `ProfilePoint`) live in `$lib/api/client.ts` — import from there
- Chart.svelte exposes `getInstance()` for attaching ECharts event handlers (getZr click, series click)
- ECharts `convertFromPixel('grid', [offsetX, offsetY])` converts pixel click coords to data coordinates
- Use `getZr().on('click')` with `!event.target` check for blank area clicks (add points)
- Use `chart.on('click')` with `componentType === 'series'` for clicking on data points
- Svelte 5 `$props()` values referenced outside reactive contexts trigger `state_referenced_locally` warning — extract to plain `const` variables to make intent clear
- WebSocket messages are polymorphic: check for `msg.telemetry` vs `msg.autotune` key to determine type — parse as generic JSON first
- Autotune API endpoints: start/stop/apply require auth (POST), status/results latest are public (GET); API client namespace is `autotune` in client.ts
- Vitest `.svelte.ts` testing requires: `svelte()` plugin + `resolve.conditions: ['browser']` in vitest.config.ts; import with `.svelte.js` extension; mock `fetch` before module imports; reset `$state` in `afterEach`
- Inline tables in init_db use CREATE TABLE IF NOT EXISTS + direct sqlx::query (no migration file needed for simple tables)
- PUT handlers returning 204 use `StatusCode::NO_CONTENT.into_response()`
- `$effect` cleanup pattern works for intervals: return `() => clearInterval(id)` from the effect body
- ECharts markLine with `yAxis: value` creates a horizontal line (vs `xAxis: value` for vertical landmark lines)
- Use spread operator `...(condition ? [item] : [])` to conditionally include series in the ECharts series array
- Import `.svelte.ts` modules in tests using `.svelte.js` extension (TypeScript convention)
- Pure functions extracted from reactive stores are the easiest to unit test — prefer this pattern for core logic
- SQLite ALTER TABLE doesn't support IF NOT EXISTS — wrap each ALTER in a separate query that silently fails if column exists, or check sqlite_master first
